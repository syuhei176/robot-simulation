## 1. 背景とゴール

- リポジトリは既存の **Microbots RL**（Three.js + TypeScript、物理もRLも自前）。
  `AGENTS.md` の本来ゴールは **6個の同一モジュールで 蛇型⇄四足 に再構成するロボット**。
- このスレッドの問いの流れ:
  1. 家の中を動く省電力ロボ（太陽光1週間・カメラ・BLE/WiFi）から出発
  2. 監視に最適な形を発散（分散静止ノード/天井レール/ヘリウム浮遊…）→ 一旦保留
  3. **蛇型ロボットに決定**、まず予算を確認
  4. 省電力の駆動・軽量化を検討 →「**軽量・床這いベースのまま、複雑な形で階段を登れないか**」
  5. → **シミュレーションで形を探索し、必要モーターを逆算**する方針で着手（本メモ）

## 2. 確定した設計インサイト（物理）

- **階段登りはエネルギー問題ではない。** 位置エネルギー `m·g·h` は微小（後述：13段で約0.006 Wh）。
  支配項は (a) **持ち上げた体を支える保持トルク**、(b) **段の縁を越える瞬間トルク**。
- **軽量化は線形〜超線形に効く。** 保持トルク・摩擦・揚力はすべて質量に比例。さらに
  サーボの**等級しきい値を跨ぐと価格が“段”で跳ねる**ので、軽さは金額に超線形で効く。
- **「動かす」と「保持する」を分離する。** 非バックドライブ（ウォームギア等）/ ブレーキ /
  EPM を使えば、休止中の保持は **0W**。サーボの最大の浪費（静止保持電流）を消せる。
- **歩容で重力を床に預ける。** 平面の床這いなら体重は床が支え、保持トルクはほぼ不要。
  電力を食うのは「持ち上げる瞬間（階段・段差）」だけ。**複雑な形は“支えを階段の幾何に
  逃がす”ために使う**（自由度を増やすためではない＝そこを外すと重く非力になるだけ）。
- **太陽光1週間は蛇型では非現実的。** 多関節の保持で電力を食う。間欠運用（動く→脱力で
  寝かす→動く）前提。連続稼働は数十分〜1時間／充電オーダー。
- 参考：脱・蛇型の階段攻略の有力形は **wheg/脚車ハイブリッド**（平地は転がり省電力、段は
  縁を掻いて登る）や **突っ張りインチワーム**。Salto（UC Berkeley）等の跳躍は「エネルギーは
  安い／狙って崩れず跳ぶ制御が高い」で、監視用途には不向きと整理済み。

## 4. モデルの仮定（Codex が把握すべき前提）

- 矢状面 **2D**（x: 前後・水平, z: 上下, 関節軸は紙面垂直 y）。回転は x-z 平面に閉じる。
- 階段一段を「lift-off 点から `(forward, rise)` へ前端を届かせる」**抽象**でモデル化。
  デフォルト段：`rise=0.18m, forward=0.10m`（やや急な住宅階段、弦 20.6cm）。
- **持ち上げ部 = 先端側の k リンク**。k は「弦長の1.25倍以上の連鎖長を確保できる最小本数」。
  残りの胴体は床に支持され、その関節の保持トルク ≈ 0 と仮定（＝床這いベース）。
- 評価する2姿勢:
  - **円弧リーチ**（効率歩容）：lift-off を立てて前へ被せる → 平常運用の低トルク。
  - **水平片持ち**（最悪・上界）：腕を真横に伸ばし切る → **設計荷重**。
- **設計荷重 = 最悪姿勢トルク × 安全率1.5** → これを満たす最小ストールトルクのサーボを選定。
- サーボのストールトルクは概算（XL330=0.4 / MG996R=1.0 / XL430=1.5 / STS3215=2.94 / XM430=3.5 N·m）。

### 既知の限界（重要）

- **静的保持トルクのみ**。動的ピーク（縁越え加速・着地衝撃）は未計上 → 実値は 1.5〜2倍に
  膨らみ得る（安全率はその保険）。
- **グリップ（滑り止め摩擦）は未モデル**。段で滑らない条件は別要件。
- 段の幾何（蹴上げ・踏面・残り胴体の干渉）は簡略化した抽象。

## 5. 主要な結果

実行: `pnpm motor-sizing`

### 基準形状（90cm・1.0kg・8リンク、段 rise18cm/forward10cm）

- 持ち上げ 3リンク（先端側）、持ち上げ質量 375g、到達検証 tip≈(10.5, 18.9)cm（目標(10,18)）OK。
- 関節別（円弧姿勢）: lift-off ≈ **0.021** / 関節1 = **0.169（ピーク）** / 関節2 = 0.069 N·m。
  → ピークは **中間関節**に集中（lift-off は立つので腕が短く≈0）。
- 3姿勢ピーク: 平面 ≈0 / 円弧 **0.169** / 水平片持ち **0.621** N·m（≒ 6.3 kg·cm）。
- 設計荷重 0.621×1.5 = **0.931 N·m** → **MG996R（1.0 N·m, ¥600）**で足りる。7関節で ≈¥4,200。
  効率歩容ではストールの **17%以下**で常用 → 低発熱・低消費。
- 登坂エネルギー: 1段 1.77J、**13段で 22.9J = 0.0064 Wh**（ほぼゼロ）。

### 形状スイープ（総長0.9m固定、設計荷重=最悪×1.5でサーボ選定）

- **0.6kg**: どの形も MG996R(¥600)。
- **1.0kg**: n=6/8/12 は ¥600、**n=4 だけ STS3215(¥2,500)**。
- **1.8kg**: n=4 → XM430 **¥25,000**、n=6 → XL430 ¥7,000、n=8 → STS3215 ¥2,500、n=12 → XL430 ¥7,000。

### 設計指針（数値が指す結論）

- **総質量 ≤ 1kg**（90cm級）に抑えると安いサーボ圏に留まれる。
- **6〜12リンクが甘い所**（ピーク低・個数も妥当）。**n=4 は罠**（荷重集中で高価サーボに追い込まれる）。
- **賢い歩容（急に立てて被せる）**で常用 <20%負荷 → 非バックドライブ/ブレーキとの相性も最良。
- 負荷は中間関節に集中するので、**中間だけ強いサーボ**にする非一律構成が有効（未数値化）。

## 6. Phase C 初版：Rapier 動的シミュレーション

目的：静的保持に **動的ピーク（縁越え加速・着地衝撃）** と **接触/滑り** を上乗せし、
実際の必要トルクとグリップ要件を出す。初版は `pnpm stair-dynamics` で実行できる。

実装内容:

1. Rapier world を作成（重力、地面、階段コライダ、Nリンクチェーン）。
2. 各リンクを cuboid 剛体、隣接リンクを revolute joint で接続。
3. scripted keyframe gait で「持ち上げ → 縁越え → 踏面へ配置」を行う。
4. Rapier の joint motor で姿勢追従し、同じ PD 式から要求トルクをログする。
5. 接触 normal impulse / dt から接触荷重ピークを出す。
6. 配置後に小さな水平牽引プローブを入れ、`牽引力 / 総法線荷重` から近似 μ 要求を出す。

基準実行（90cm・1.0kg・8リンク、rise18cm/forward10cm、μ=0.8）:

- 持ち上げ 3リンク。
- 静的円弧ピーク **0.169 N·m**。
- 動的PD要求ピーク **0.501 N·m**（joint=5、静的比 **2.96x**）。
- 接触 normal ピーク **9.5N**。
- 牽引プローブによる μ 要求 ≈ **0.28**。
- head tip final ≈ **(7.7cm, 19.3cm)**、success=true。

実装メモ:

- 既存の型ストリップ流儀で OK（`node scripts/xxx.ts`、相対 import は `.ts` 拡張子付き）。
- `rapier3d-compat` は **`await RAPIER.init()`** が必要（WASM 非同期）。
- Rapier JS の公開型では joint motor の実適用トルクを直接読めないため、現状は「同じ PD 式の要求トルク」をログしている。
- 摩擦は `contactTangentImpulse` がこのシナリオで 0 のため、牽引プローブの `F/N` を近似 μ として使う。
- 既存 `src/sim`（平面RL）は触らない。3D系は `src/sim3d/` に隔離する方針。
- Rapier で精度不足なら **MuJoCo（Python）** へ切替も選択肢（接触が高精度）。

## 6.6 完全階段 physical attempt の結果

ユーザー指摘どおり、前の replay は理想的すぎた。そのため、同じキネマティック軌道を目標にしつつ、
実体は Rapier で以下の制約を受けるようにした。

- **トルク**: 各 revolute joint に手動 PD トルクを入れ、`maxTorqueNm` で clamp。
- **落下/干渉**: 剛体リンクと階段 collider の接触を Rapier が解く。追従できなければ実際にぶつかる/落ちる。
- **滑り/牽引不足**: 前進用の牽引力は `μmg` を上限にして、μ不足なら目標位置へ届かない。
- **必要 μ/clearance**: replay 後に `stair-feasibility.ts` でフレームごとの診断も付与。

基準 physical attempt（`τ上限=0.25 N·m`, `μ=0.60`）:

- 状態: failed。
- 最終 head tip は約 **(-0.1cm, 1.4cm)**。最上段ではなく、ほぼ床付近で失敗。
- motor demand peak は約 **10.58 N·m**、applied peak は cap の **0.25 N·m**。
- saturation steps は **462**。
- 初回破綻は約 **2.38s**。
- failure frames: torque=462 / fall-or-collision=155 / slip=194 / high-mu=169。
- min clearance は約 **-181 mm**。
- max unsupported span は **90 cm**（全身が支持を失う局面がある）。

解釈:

- `τ上限=0.25 N·m` では明確にトルク不足。見た目も理想通りには登らず、実際に崩れる。
- 高トルク/高摩擦にしても、現状の目標軌道は段の立ち上がりへぶつかるため成功しない。
  つまり問題は「モーターだけ」ではなく、**衝突を避けた clearance-aware 軌道**と
  **接触を使う登攀コントローラ**がまだ無いこと。
- 次の段階は、段差手前で前端を先に持ち上げる軌道生成、支持点の切替、body を引き上げる
  traction/anchor 戦略を物理ループ内で作ること。

## 6.8 SG90 検討の結論（静的＋ゆっくり歩容スイープ）

**静的（`pnpm motor-sizing`）**: 90cm・1.0kg はSG90に **5.3倍不足**。SG90 に乗せるには
(A) 実階段18cm維持なら総質量 **≤0.19kg**、(B) 幾何相似ダウンスケール（トイ階段）なら **s=0.6＝54cm/0.22kg級**。
（トルク ∝ s⁴ なので縮小は激効き。ただしBは縮小した段で、実18cm階段ではない）

**動的（`pnpm stair-dynamics` の SG90 ゆっくり歩容スイープ）— 重要な訂正**:

- 動的要求ピークは gait×1〜12 で **完全に平坦** ＝ 加速由来ではなく **準静的な保持トルク**。→ **「ゆっくり化」では下がらない**（ケイデンスは誤ったレバー）。効くのは質量・姿勢・支持の取り方。
- アーティファクト2つ（**固定3N牽引プローブ**が軽量機に過大／**生成直後の整定スパイク** t≈0.04s）を除くと、歩容本体の真の単段ピークは **A:0.115 N·m=SG90の0.65x（収まる）／B:0.227 N·m=1.29x（わずか超過）**。
- 追加API（`src/sim3d/stair-dynamics.ts`）: `gaitTimeScale`（歩容時間スケール）, `demandWarmupS`（整定除外）, `maxDemandTimeS`（ピーク時刻）。

| 四足 | 踏み出し(3脚支持)      | 2.6          | 3.9      | ✗    | ✗     | ✗       | ✓       | 4    |
| 四足 | 段差リーチ(rise18cm)   | 7.4          | 11.1     | ✗    | ✗     | ✗       | ✓       | 4    |

**結論**: 四足は平地立脚なら SG90 でも足りる（体重を脚で床へ預ける）。が、踏み出し・段差リーチでは STS3215 必須＝**階段では蛇と同じく強モーターが要る**。四足の旨味は「立っているだけなら安い・関節少」。

## 6.10 四足 3D 動的歩行（Rapier・IKクロール・忠実トルク）＋ viewer 統合

`src/sim3d/quadruped-dynamics.ts`。当初は内蔵位置モーター＋cap模擬だったが、中間トルク帯で非単調だった（旧 §6.11）ため **IKクロール＋忠実トルクに作り直した**（§6.12）。

- 構成: 胴(0.24×0.16×0.05m/0.52kg) ＋ 4脚(各 thigh9+shin9cm・85g/seg、hip/膝の2関節) = **合計1.2kg・8関節**。全関節 y軸 pitch。
- 歩容: **IKクロール**。足先のカートシアン軌道（立脚は胴に対し後方へ等速で掃き接地摩擦で前進、遊脚は持ち上げて戻す）を2リンクIKで hip/膝の目標角に変換。1脚ずつ遊脚で常時3脚接地。`pnpm quadruped` で実行。
- 駆動: **忠実トルク**。PDトルクを **±maxTorqueNm でハード clamp** し、作用反作用で `applyTorqueImpulse(τ·dt)` として印加（陽解法をサブステップで安定化、ギア摩擦相当の受動ダンピング併用）。
- 横安定化: 脚が pitch のみで横(ロール/ヨー)を制御できないため、横バランスは別機構前提としてトランクのロール/ヨーだけ外部トルクで安定化（矢状面の歩行・トルク充足を切り出す）。
- 結果(1.2kg): **全機種 ~26〜29cm で歩く**（胴速度レギュレータで足が世界接地点に固定され滑らない＝過走なし）。飽和%は強capほど低下(100%→10%)。重さを上げると弱capから脱落（1.6kgでSG90転倒）＝重さ応答。
- viewer: `stair.html` の機構=四足で動的歩行を再生。**総質量スライダー(0.8〜2.5kg, 既定1.2)**追加。関節マーカー球(hip/膝=アンバー)・追従カメラ・モーター選択でcap自動設定。
- **限界**: 平地のみ。**段差/階段の登坂は未対応**。横安定化は外部前提。軽い脚＋強capは暴走しやすいので総質量下限は1.2kg（それ未満は脚慣性不足で強servoが launch）。

## 6.11 中間サーボ追加と軽量化検討（四足・平地歩行）

「四足は STS3215 でないと前進しない？」「軽量化で安servoは歩ける？」の検討。

**中間サーボ追加**: `SELECTABLE_SERVO_IDS` に **MG996R**(10kg·cm/¥600) を追加（5機種化）。viewer の `<select>` は
`SELECTABLE_SERVO_IDS` から動的生成に変更（`stair.html` のハードコード重複を排除）。

**軽量化は効く（静的モデル＝信頼できる）**: 平地歩行の律速は **踏み出し(3脚支持)** の保持トルク。
これは質量に線形（payload=mg/3）なので、各サーボが満たせる上限総質量（設計荷重=最悪×1.5）:

| サーボ  | ストール | 上限総質量  |
| ------- | -------- | ----------- |
| SG90    | 1.8kg·cm | **≤0.47kg** |
| MG90S   | 2.2      | ≤0.57kg     |
| SCS0009 | 2.3      | ≤0.60kg     |
| MG996R  | 10.2     | ≤2.65kg     |
| STS3215 | 30       | ≤7.80kg     |

→ **1.0kg級なら MG996R が正解サイズ**（設計荷重3.8kg·cm を余裕で満たし STS3215 は過剰）。SG90 で歩かせたければ **総質量≈0.47kg 以下** が要る。
この上表と総質量別の「満たす最安サーボ」は **`pnpm quadruped` の静的サーボ選定スイープ**で再現できる（動的表の下に出力）。

**（当初の問題・現在は解決）動的シムが中間トルク帯で非単調だった**: 旧実装（内蔵位置モーター＋cap模擬）では
cap→前進距離 が **非単調**（SG90/MG90S/SCS0009 失速 → MG996R も失速 → STS3215 のみ成功）で、物理的に十分な
MG996R が失速していた。原因は `driveJoint` の cap 模擬（目標角を詰める方式）が torque 上限を物理的に表現
できないこと。→ **§6.12 で IKクロール＋忠実トルクに作り直して単調・重さ応答に解決済み**。静的モデル(§6.9 + 上表)は
依然サーボ選定の保守的な一次見積りとして有効（動的の効率歩容は姿勢が立っている分やや軽い要求になる）。

## 6.12 動的シムの作り直し（cap模擬 → IKクロール＋忠実トルク）

§6.11 の非単調を解消するため、動的歩行を物理忠実に作り直した（`src/sim3d/quadruped-dynamics.ts`, `motor.mode='torque'`）。

**何が問題だったか**: 内蔵位置モーターは安定だが cap 模擬（目標角を詰める）が torque 上限を表現できず、
cap→歩行性能が非単調（弱capが強capより歩く逆転すら起きる）。一方、素朴な手動トルクは試行で連続的に失敗:

1. **符号** — `addTorque` 用の符号が位置モーター規約(-1)のままで正帰還＝発散。pitchAboutY は +y 回転角なので child へ +τ が正。
2. **永続力** — Rapier の `addTorque` は reset まで毎ステップ累積する → `applyTorqueImpulse(τ·dt)` で1ステップだけ与える。
3. **陽解法の不安定** — 軽い脚(I≈1e-4)＋剛い PD で発散 → **サブステップ**（physicsDt=dt/substeps で PD 再計算）。
4. **横転** — 脚が pitch のみで横バランス不能（1脚上げ時に重心が支持三角形の境界）→ **トランクのロール/ヨーを外部安定化**（横は別機構前提）。
5. **足の滑り/skating** — 関節角直接指定だと足が滑る → **足先カートシアン軌道＋2リンクIK**で立脚足を接地保持。
6. **強capの暴走/launch** — 脚が軽すぎ(I≈1e-4)て強cap(2.94N·m→α≈3万rad/s²)で暴走 → **脚 segMass を現実的に(85g)** ＝関節にサーボ＋ギア反映慣性がある実機に寄せる。受動ダンピングも併用。
7. **強capの過走/足の滑り** — cap が高い機種は接地足が後方へ滑って歩容速度を超えて過走（STS3215 が他の倍進む）→
   **胴速度レギュレータ**: 基準前進量 `vBody·t`（=stride/(stanceDuty·period)）と実前進量の差を足先 fx に足し込み、
   足を世界の接地点へ寄せる（速すぎれば足を前へ→推力減で減速、遅れれば足を後ろへ→推力増）。±stride で clamp。

**結果**: 総質量1.2kg で **全機種 ~26〜29cm に密集**（SG90 26 / MG90S 26 / SCS0009 26 / MG996R 29 / STS3215 29）＝
レギュレータで足が滑らず歩容速度に収束（強capの過走が消えた）。飽和%は強capほど低下(100%→10%)。
重さを上げると弱capから脱落（**1.6kg で SG90 転倒**）＝重さ応答。viewer に総質量スライダー(0.8〜2.5kg, 既定1.2)を追加し「動く重さ」を探索できる。

**残る前提・限界**: 横安定化は外部前提（脚に横DOFが無い形態の制約）。スライダーは0.8kgまで下げられるが、
**1.2kg未満では強cap(MG996R/STS3215)が過剰で暴走/launch**する（弱capは0.8kgでも歩く）。サブ1.2kgのサーボ選定は
静的モデル(§6.11)が信頼できる道具。
動的の効率歩容（脚が立ち気味）は静的の最悪姿勢より要求が軽く、絶対しきい値は静的(§6.11)とずれる（向きは一致）。
更に忠実化するなら関節にサーボ質量を点荷重で置く／MuJoCo。

## 7. 開いている判断・宿題

- 物理エンジン：**Rapier 採用**（ブラウザ/TSに地続き）。動的接触で精度不足なら MuJoCo を検討。
- 蛇：完全階段の**物理 attempt 表示**と feasibility 診断は実装済み。だが「目標追従型」で、成功する登攀コントローラではない。replay は段の立ち上がりに干渉 → 次は **clearance-aware 軌道**＋接触選択型コントローラ。
- 四足：**平地 IKクロール歩行を忠実トルクで実装済み（3D Rapier・歩く・単調・重さ応答, §6.12）**。次は **段差/階段の登坂**（前脚を次段へ掛ける gait、3脚支持を保つ重心移動）。蛇と共通の「接触を使う登攀」課題。
- トルクcapは **忠実化済み**（±cap ハードclamp＋applyTorqueImpulse＋サブステップ）。残課題: 横バランスは外部安定化に依存（脚に横DOF追加 or 動的バランス制御）、より忠実にするなら関節にサーボ質量を点荷重で置く／MuJoCo。
- 四足の歩容は手作り IKクロール。**RL で歩容/登攀を学習**は morphology/torque cap/接触/転倒が揃ったので着手可能。
- 「中間関節だけ強い」**非一律サーボ構成**の数値詰め（未着手）。**グリップ機構**（滑り止め）の設計・モデル化（未着手）。
- 監視用途に戻る場合の上位形（分散静止ノード等）は別トラックとして保留。
